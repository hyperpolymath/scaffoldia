`developer-ecosystem/scaffoldia/registry/` — the GitLab-lineage scaffoldia,
which is an *implemented* Haskell CLI + Nickel builder that this (GitHub)
repository's own blocker list previously ignored ("composer implementation
not yet decided"). Fork resolution, recorded in
`.machine_readable/descriptiles/STATE.a2ml`:

* *This* repo is the canonical design surface and the future consumer of the
  spine's contracts (`ARCHETYPE.a2ml` / `VARIANT.a2ml`, rsr-template-repo

* The GitLab-lineage engine stays *dormant* — not deleted, not developed in
  parallel. Its registry entries live here as **data** informing pack
  definitions; its Haskell/Nickel engine is one candidate for the Phase 1
  composer ADR, which must weigh it explicitly rather than ignore it.

Two harvest caveats, flagged not fixed (no automated licence edits — ever):

* `julia/package.ncl` declares `license = "PMPL-1.0-or-later"` — outside the
  Palimpsest carve-out (exactly three repos), so a scaffold minting that
  header would violate the licence policy. Owner review needed before any
  consumer uses it.
* Entries embed `jonathan.jewell@open.ac.uk` authorship defaults; consumers
  must take identity from `just init` answers, never from harvested data.

The *profile* composition axis (previously a name with no schema anywhere in
this repo) is resolved by ADR-0003: profile = the capability set of
standards `.machine_readable/template-capability-gates.toml` — a preset name
or a direct capability list, exactly what `rsr-profile.a2ml` declares.
